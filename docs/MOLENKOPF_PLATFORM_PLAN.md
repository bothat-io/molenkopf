# Molenkopf Platform Plan

Date: 2026-06-18

Status: historical target architecture. Use `ROADMAP.md` for current release work.

This is the product and architecture plan for turning Molenkopf into a team
gateway for providers, plugins, compression, memory, and employee access.
Milestones and verification live in
[MOLENKOPF_PLATFORM_MILESTONES.md](MOLENKOPF_PLATFORM_MILESTONES.md).
Concrete implementation packages live in

The canonical provider and team configuration target is

## Target

Molenkopf becomes a controlled provider and plugin gateway. Employees and local
agents use Molenkopf as their LLM endpoint. They receive Molenkopf credentials,
not upstream provider credentials.

```text
employee / CI / local agent / CLI runtime
  -> Molenkopf token
  -> auth, scopes, provider policy, plugin policy
  -> core safety
  -> optional compression and memory plugins
  -> provider adapter
  -> OpenAI / Anthropic / local model / CLI runtime
```

## Current Reality

- Provider profiles, scoped API keys, project attribution, team allowlists, and
  manual/distributed routing exist.
- Agent rows are routing metadata; employee/user auth and API keys are backed by
  the local identity store.
- `__molenkopf/*` local APIs are protected once an admin exists; non-loopback
  source binds require an explicit public-bind opt-in.
- Selected provider profiles inject server-side credentials at the forwarding
  boundary; local/no-auth profiles strip incoming client credentials.
- Plugin toggles exist for optional plugins; safety, storage, and routing
  invariants are core behavior, not dashboard plugins.
- Compression works on large JSON strings, but accounting must separate real
  compression savings from redaction or JSON serialization changes.
- The graph page is derived from redacted transferred text and safe metadata;
  Obsidian vault export is still deferred.
- The dashboard is a local operator console; final product naming and wider
  employee operations polish remain release work.

## Non-Negotiables

- Provider keys stay in env, vault, or a local secret store.
- Employees get Molenkopf tokens, not OpenAI or Anthropic keys.
- Molenkopf tokens are generated by Molenkopf, shown once, stored only as hashes,
  and revocable.
- Full prompts, full responses, cookies, authorization headers, and raw provider
  keys cannot be represented in audit/dashboard types.
- Plugins are runtime capabilities with hooks, permissions, and scoped data.
- Obsidian writes require dry-run first and path guards before apply.
- Core and proxy keep Node built-ins only.
- Dashboard dependencies stay isolated in `packages/dashboard`.

## Ten Planning Slices

1. Provider routing:
   unify provider contracts, inject env credentials only at forwarding, add
   health, failover, budget, and per-agent provider selection.

2. Agent auth:
   replace drafts with users, agents, token hashes, scopes, revocation, last
   seen state, provider binding, and plugin policy.

3. Plugin runtime:
   add descriptors with hook phases, permissions, workspace kind, data scopes,
   lifecycle events, and toggle policies for optional runtime extensions.

4. Compression:
   report real compression savings only, commit retrieval artifacts only after
   compression succeeds, and separate compression from redaction/serialization.

5. Memory and Obsidian:
   derive semantic memory from safe transferred text, store bounded summaries
   and source refs, then dry-run/apply Markdown writes to an explicit vault root.

6. Dashboard:
   grow into a wide operator console with Routing, People/Agents, Plugins,
   Requests Explorer, Audit, Memory, and Settings.

7. API surface:
   make Molenkopf usable as an OpenAI-compatible base URL with Molenkopf tokens,
   then add protocol adapters for Anthropic and local model providers.

8. Security and audit:
   protect local APIs, redact query paths before audit write, add retention and
   bounded pagination, and test that no secret or raw body leaks.

9. Employee operations:
   admin creates providers, profiles, employee/agent tokens, setup snippets,
   rotation/revoke workflows, and diagnostic views.

10. Build program:
    implement by stable vertical slices with tests at each boundary, not by
    adding a large unverified framework.

## Core Contracts

### Molenkopf Token

```text
token: mk_<tokenId>_<secret>
stored: tokenId, sha256(secret), scopes, owner, providerProfileId,
        pluginPolicyId, createdAt, expiresAt, revokedAt, lastUsedAt
```

Scopes:

```text
proxy:use
control:read
audit:read
events:read
agents:manage
providers:manage
providers:select
plugins:toggle
memory:read
memory:write
obsidian:dry-run
obsidian:apply
```

### Provider Profile

```text
id
kind: openai-compatible | anthropic | local-openai | cli-codex | cli-claude
mode: openai-compatible | anthropic-native | anthropic-openai-compatible
target
credentialRef: env:OPENAI_API_KEY | env:ANTHROPIC_API_KEY | none
auth.scheme
allowedModels
health
failover
budget
```

### Plugin Descriptor

```text
id
category
hooks
permissions
dataScopes
workspace
togglePolicy
configSchema
```

Toggle rule: if disabling a feature must not be allowed for safety, the feature
belongs in Core instead of the plugin catalog.

First admin access is bootstrapped by a CLI command that prints the raw token
once and persists only the hash.

## Employee Setup Target

Later target for OpenAI-compatible clients:

```text
OPENAI_BASE_URL=http://molenkopf.example/v1
OPENAI_API_KEY=replace-with-molenkopf-token
```

Molenkopf interprets the token. Upstream provider credentials stay server-side.

## Do Not Build First

- No remote plugin marketplace.
- No accidental public bind without explicit opt-in.
- No raw Obsidian writes without dry-run.
- No response-body storage until a separate safe response contract exists.
- No generic plugin framework before built-in descriptors and tests exist.
- No employee distribution based on provider keys.
